# Worktree Workflow

## Before Starting Any Task

**Never make changes directly on `main`.** Create an isolated git worktree on a feature branch:

```bash
mkdir -p ../cloudish-worktrees
git worktree add ../cloudish-worktrees/cloudish-<feature-slug> -b feat/<feature-slug>
cd ../cloudish-worktrees/cloudish-<feature-slug>
npm install
```

- Worktrees always go inside `../cloudish-worktrees/` (sibling of the main repo), never as siblings of the repo root.
- Always run `npm install` — each worktree has its own `node_modules`.
- All edits, installs, and commits happen inside the worktree, not in the main working tree.

## Before Committing

- Run `npm run lint` before every commit.
- For tasks touching Tailwind classes, run a class-level lint check (`suggestCanonicalClasses` warnings).
- Apply Tailwind class suggestions only when they are low-risk and don't change layout intent or accessibility semantics.

## Pull Requests

- **Every completed task requires a PR**, even small changes — but ask the user if they want a PR or a direct merge to main.
- Agent can merge without human review.
- PR description must include: what changed, why it changed, how it was verified.
- PR body must include: **"🤖 Generated by Claude Code (AI)"** at top or bottom.
- Add PR comments for implementation details, tradeoffs, follow-up work, or verification notes worth preserving.
- Include screenshots for UI, documentation, or visual changes. Use before/after when helpful.
- **PRs do not require approval — merge directly after creation.**
- If self-approval is rejected by GitHub, leave a PR comment noting that approval was attempted and rejected.
- After merging, update local `main` to the merge commit.

## Cleanup After Merge

Immediately after the PR merges:

```bash
git worktree remove ../cloudish-worktrees/cloudish-<feature-slug>
git branch -d feat/<feature-slug>
```

Do not leave stale worktrees. A worktree only exists while the PR is in-flight.

## Temporary File Cleanup

Before considering a task complete, delete all temporary files created during development:
- Automation scripts (`.mjs`, `.sh`, one-off Node scripts for testing/debugging)
- Screenshot/export artifacts (`.png`, `.html`, `.json` snapshots for debugging)
- Test/debug outputs (logs, data dumps, temporary exports)

A completed PR should contain only intentional, meaningful changes — not development scaffolding.

## Coordination

If another agent's worktree already exists for the same feature, coordinate with the human before creating a new one. Do not overwrite in-progress work.
